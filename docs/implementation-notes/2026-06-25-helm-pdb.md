# 2026-06-25 — Helm PodDisruptionBudget (app + worker)

**Commit:** `<sha>` infra(helm): add PodDisruptionBudget to app + worker

## Design

The chart shipped an app Deployment + HPA (min 2 / max 10) and a worker
Deployment (prod: 2 replicas) but **no PodDisruptionBudget** — so a
single voluntary disruption (node drain on a k8s upgrade, cluster
autoscaler scale-in, kube-system node compaction) could evict every pod
at once. The HPA can't catch that: it reacts to load, not to disruption
velocity.

`templates/pdb.yaml` emits two `policy/v1` PDBs (app + worker), each
gated by `pdb.{app,worker}.enabled`, selecting pods via the shared
`inflect.selectorLabels` helper with the matching component so the PDB
selector can never drift from its Deployment's pod selector.

| `docs/deployment.md` | PDB section (policy, per-env, verify) |

## Decisions

- **`maxUnavailable: 1`, not `minAvailable: N`.** With HPA-driven
  replicas a percentage `minAvailable` plays poorly during scale-down —
  it can block a legitimate HPA shrink. `maxUnavailable: 1` keeps ≥1 pod
  available even at the `minReplicas: 2` floor and never blocks scale-down.
- **Staging carve-out (disabled by default).** A single-replica
  Deployment under `maxUnavailable: 1` yields `ALLOWED DISRUPTIONS: 0`,
  so **every node drain blocks indefinitely**. Staging runs
  single-replica today, so it must inherit `enabled: false`; the ratchet
  also asserts prod's HPA `minReplicas >= 2` so prod can't regress into
  the same trap.
- **`policy/v1`** — stable since k8s 1.21; chart `kubeVersion` floor is
  already `>= 1.28.0`, so no bump needed.
- **Per-component gates** — an operator testing a single-replica cluster
  can disable one component without losing the other's protection.

## Operator note

Verify enforcement with `kubectl get pdb -n inflect-production`
(`ALLOWED DISRUPTIONS: 1` = healthy). During a `kubectl drain`, the PDB
makes the eviction API take down at most one app pod at a time and wait
for a Ready replacement before the next — instead of all at once.
