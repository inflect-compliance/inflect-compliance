# 2026-06-13 — RQ3-10 — Risk Board page

**Commit:** `pending` (branch `claude/rq3-10-board-page`)

## Design

The risk dashboard (RQ3-9) is the daily analyst surface — every signal, every
chart, every nudge. The board page is its high-altitude sibling: the five
questions an exec actually asks, stripped of operational noise. The contract is
deliberately small.

```
┌────────────────────────────────────────────┐
│  1. Position    — Annual loss exposure     │  ← portfolioP80 + tone band
├────────────────────────────────────────────┤
│  2. Appetite    — Within / approaching /   │  ← from RQ2-6 status
│                   breached + ceiling       │
├────────────────────────────────────────────┤
│  3. Top contributors — top 5 by tail ALE   │  ← topByAle ⨯ tailByRisk
├────────────────────────────────────────────┤
│  4. Best-value controls — top 3 by ROI     │  ← RQ3-8 /controls/best-value
├────────────────────────────────────────────┤
│  5. Hygiene     — X / Y stale, %, all-fresh │  ← from RQ2-8 staleness
└────────────────────────────────────────────┘
```

### No new endpoint

The board page consumes the SAME `/risks/dashboard` orchestrator the dashboard
does (RQ3-9 prerequisite) plus the RQ3-8 `/controls/best-value` endpoint. Two
SWR calls instead of seven, both cached, and the orchestrator already does the
parallel fan-out server-side. A parallel board-specific orchestrator would have
been a second source of truth waiting to drift — the ratchet locks the
single-source contract in.

### Honest-null contract

Each card carries its own typed empty-state copy:

- **Position** — "Not quantified yet — run a portfolio simulation on the
  operational dashboard." (Never a fabricated €0.)
- **Appetite** — "Set a portfolio loss ceiling on the appetite configuration
  page to anchor the board's tolerance." (NONE status is the empty state.)
- **Top contributors** — "No quantified risks yet — once a risk carries SLE × ARO
  or FAIR inputs, the contributors list ranks itself."
- **Best-value** — "No control yet carries a price + an effectiveness signal +
  a quantified linked risk. Price the highest-leverage controls first." (Same
  shape as the RQ3-8 dashboard rail's empty state.)
- **Hygiene** — "Every assessment is current." (Distinct affordance for the
  staleCount=0 path; not silent.)

A board pack with "Not quantified yet" is more honest — and more actionable —
than one with a fabricated zero.

## Decisions

- **Five sections, no fewer.** The ratchet locks each `board-*-card` testid so
  a "minor polish" PR that drops a section fails CI. Adding a sixth is fine —
  the contract is "at least these five".
- **Top contributors capped at 5, best-value at 3.** The dashboard already
  takes 5 best-value rows for its rail; the board view is denser still, so 3 is
  the right scale. Both caps are pinned by the ratchet.
- **Reuse `DashboardLayout`.** The board page is layout-identical to the
  dashboard (page header + stacked cards), just with different content. Sharing
  the shell keeps the print/PDF-export pipeline consistent for both surfaces.
- **`<KPIStat tone="…">` instead of inline tone classes.** The Position card's
  headline tone tracks the appetite verdict (success → success, approaching →
  attention, breached → critical). This is the canonical KPI primitive — no
  bespoke colour ladder, no reaching back for `getStatusTone`.
- **`SWRConfig` in the rendered test.** Mirrors the RQ3-9 pattern — fresh cache
  provider per render so two `mockFetch` calls in different tests don't see
  each other's data.
- **No navigation entry added.** The existing top-level "Board" sidebar entry
  is the org-wide dashboard at `/dashboard`. The risk-specific board lives at
  `/risks/board` and is reached from the risk dashboard's actions slot
  ("Operational dashboard" goes the other way; future PRs can add the
  reciprocal link from the dashboard). Scope-tight on purpose.
