**Commit:** `<pending> perf(controls,tasks): #102 items 2-4 — TaskKeySequence, dashboard groupBy, Task indexes`

Three independently-scoped perf follow-ups from #102 (deferred from
the #101 controls/tasks page-load pass), bundled — all server-side,
no UI surface.

## Item 2 — `TaskKeySequence` (race-free `TSK-N` keys)

`WorkItemRepository.create` derived the next work-item key from
`db.task.count({ where: { tenantId } })` — which (a) raced the
unique `[tenantId, key]` index under concurrent imports and (b)
scaled linearly with tenant size.

New tenant-scoped table `TaskKeySequence { tenantId @id, lastValue }`.
`create` now does `db.taskKeySequence.upsert({ update: { lastValue:
{ increment: 1 } } })` — Prisma compiles this to a native
`INSERT … ON CONFLICT DO UPDATE`, so the increment is atomic at the
DB level. The migration creates the table, **backfills** each
tenant's counter from `MAX(TSK-N)` of existing keys (before RLS is
enabled), then applies the canonical Class-A RLS policy trio.

`tests/integration/task-key-sequence.test.ts` proves 20 fully
concurrent creates yield a contiguous, collision-free `TSK-1 … TSK-20`.

## Item 3 — `getControlDashboard` groupBy refactor

The dashboard `findMany`'d **every** control WITH its full
`controlTasks` array (and an unused `_count`) and reduced in JS —
loading the whole control × task graph to produce a handful of
counts. Replaced with a `Promise.all` fan-out of indexed aggregates:
`groupBy` for status + applicability distributions, `count` for
implemented / due-soon / overdue, and — for the top-owners roll-up,
which Prisma can't express as a single cross-relation `groupBy` —
`groupBy(['controlId'])` on open tasks folded into a thin
`control → owner` projection in JS (no `$queryRaw` needed). Return
shape and every derived figure are unchanged.

## Item 4 — Task index review

- Drop `@@index([tenantId, dueAt])` — a strict prefix of the
  existing `[tenantId, dueAt, status]`.
- Add `@@index([tenantId, controlId, status])` — the control-detail
  Tasks tab filters by both `controlId` and `status`.

## Decisions

- **Backfill before RLS.** The migration seeds `TaskKeySequence`
  rows before `ENABLE ROW LEVEL SECURITY`, so the cross-tenant
  `INSERT … SELECT` isn't filtered by the tenant-isolation policy.
- **No `$queryRaw` for top-owners.** #102 suggested raw SQL because
  Prisma can't `groupBy` a cross-relation. Grouping open tasks by
  `controlId` and folding into owners over a thin control list is
  just as cheap (no wide-row load) and keeps the query type-safe.
- **`getControlDashboard` had no unit test** before this change.
  The refactor is mechanically behaviour-preserving (every output
  field reviewed against the original) and the endpoint
  (`/api/t/:slug/controls/dashboard`) is covered by the controls
  E2E; a dedicated test was out of scope for the perf pass.
