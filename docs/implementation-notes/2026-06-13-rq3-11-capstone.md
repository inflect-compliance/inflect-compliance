# 2026-06-13 — RQ3-11 — Capstone (audit + architecture record)

**Commit:** `pending` (branch `claude/rq3-11-capstone`)

## Design

The RQ3 cohort shipped 15 substantive PRs across three days
(2026-06-11 → 2026-06-13). Each PR landed an implementation note
under `docs/implementation-notes/` and a structural ratchet under
`tests/guards/`. The notes are individually readable, but the cohort
as a whole was indexed nowhere — a new contributor landing post-RQ3
would have to grep + read every note to reconstruct the shape.

This PR is doc-only. It adds:

1. **`docs/rq3-roadmap-complete.md`** — the canonical architecture
   record for the RQ3 wave. Thesis, shape, full cohort table (16
   rows linking PR + implementation note + ratchet + closing
   issue), load-bearing decisions, extension guidance.
2. **`tests/guards/rq3-11-capstone.test.ts`** — a meta-ratchet that
   enforces the discoverability contract: every RQ3 implementation
   note AND every `rq3-*` guard test must be named in the capstone.
   A future RQ3 follow-up that ships code without threading itself
   into the capstone fails CI.
3. **CLAUDE.md entry** — a single section pointing at the capstone
   so a contributor reading the project's primary onboarding file
   finds the index in two clicks.

## Why a ratchet on a doc

The ratchet is structural, not stylistic. A future PR that lands a
new RQ3 deliverable without updating the capstone leaves a dangling
decision the index promised would be findable from one URL. The
contract is "the capstone is the index" — the ratchet enforces it.

The check is conservative. It enforces presence (every existing
note + ratchet appears somewhere in the capstone), not exact row
content. A doc-tightening PR can reorganise the table; it cannot
silently drop a row.

It is also monotone in the right direction: the capstone can
PROMISE more than exists today (e.g. it can name `rq3-ob-e-a11y.md`
before that PR lands), but it cannot promise less than exists.
Shipping the capstone alongside open downstream PRs (#1056 OB-E,
#1057 OB-F) doesn't break the ratchet — the existing notes are
named, and the future notes' rows are pre-listed.

## Decisions

- **One file, not a directory.** A `docs/rq3/` directory with one
  file per sub-roadmap would have given a deeper hierarchy but a
  worse onboarding experience. The capstone is meant to be read
  end-to-end on first contact; it links OUT to the individual
  implementation notes for depth.
- **Cohort table is the data model.** The table (PR + note + ratchet
  + closing issue per row) IS the index the ratchet enforces. Other
  sections (thesis, decisions, extension guidance) are prose and
  unenforceable; the table is the contract.
- **`rq3-monte-carlo.test.ts` acknowledged but not in the cohort
  table.** It pre-dates RQ3 (it ratchets the RQ-3 epic's Monte Carlo
  loop) and the cohort builds on it without modifying it. A short
  paragraph below the ratchet section names it so the meta-ratchet
  finds it; cohort table stays focused on RQ3-1 onward.
- **No code changes.** The capstone is a record, not a refactor.
  Cohort PRs already landed their structural assertions; this PR
  just makes them collectively discoverable.
