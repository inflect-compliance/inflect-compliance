# Internationalization (i18n) & localization

Inflect ships a **multi-locale UI** on [`next-intl`](https://next-intl.dev)
(v4, App Router, **no** URL-based routing). Locale is a per-browser
preference stored in a cookie and resolved server-side, so the first SSR
byte is already in the user's language — no flash, no client round-trip.

Shipped locales today: **English (`en`, default)** and **Bulgarian (`bg`)**.

## How it works

```
inflect_locale cookie ──► src/i18n.ts (getRequestConfig)
                             │  resolveLocale(cookie) → 'en' | 'bg'
                             │  import(`messages/<locale>.json`)
                             ▼
        getLocale() / getMessages()  (next-intl/server)
                             │
          ┌──────────────────┴───────────────────┐
   <html lang={locale}>              NextIntlClientProvider
   (src/app/layout.tsx)                (messages + locale)
                                              │
                        useTranslations() / getTranslations()
```

| Piece | File | Role |
|---|---|---|
| Supported locales + resolver | `src/lib/locale-constants.ts` | `SUPPORTED_LOCALES`, `DEFAULT_LOCALE`, `LOCALE_COOKIE`, `LOCALE_LABELS`, `resolveLocale()` (server-safe — no `'use client'`) |
| Request config | `src/i18n.ts` | Reads the cookie, loads the matching catalog. Unknown/absent → default. |
| Set-locale action | `src/lib/set-locale-action.ts` | `'use server'` — writes the cookie (coerced through `resolveLocale`, 1-year, `sameSite=lax`). |
| Switcher UI | `src/components/layout/LocaleSwitcher.tsx` | Segmented `<ToggleGroup>` in `<UserMenu>`; calls the action then `router.refresh()`. |
| Catalogs | `messages/en.json`, `messages/bg.json` | The message keysets. `en.json` is the source of truth. |

`resolveLocale()` is the single choke point: every value that could pick a
catalog (cookie, switcher input) passes through it, so a tampered cookie can
never point the dynamic `import()` at a missing file (path-traversal-safe).

## Adding / changing strings

Namespaces map 1:1 to the top-level keys in `messages/en.json` (`common`,
`nav`, `risks`, `controls`, …). The consistent pattern:

- **Server component / page:** `const t = await getTranslations('<namespace>')`
  then `t('key')`. Pages that render through a client island build a labels
  object of `t(...)` calls and pass it down as props.
- **Client component:** `const t = useTranslations('<namespace>')` then
  `t('key')`. The active locale is available via `useLocale()`.

**Every key you add to `en.json` MUST be added to `bg.json` too** (translated,
same `{placeholder}` tokens). This is enforced — see below.

## Enforcement

`tests/guardrails/i18n-completeness.test.ts` (GAP-19) fails CI on any drift
between `en.json` and every other locale: MISSING keys, ORPHAN keys, and
`{placeholder}` mismatches. Run the same check from a shell with
`node scripts/i18n-diff.mjs --check`.

`tests/guardrails/i18n-locale-infrastructure.test.ts` locks the plumbing: the
supported-locale set + resolver behaviour, a parseable catalog per locale, and
a regression guard that `src/i18n.ts` resolves the cookie (never re-pins to a
hardcoded `'en'`).

## Adding a new locale

1. Add the code to `SUPPORTED_LOCALES` + an endonym to `LOCALE_LABELS` in
   `src/lib/locale-constants.ts`.
2. Copy `messages/en.json` to `messages/<locale>.json` and translate every
   value (keep the keys + `{placeholders}` identical — the GAP-19 guard
   enforces this).
3. That's it — the switcher, `<html lang>`, and request config all read
   `SUPPORTED_LOCALES`, so the new locale appears automatically.

## Roadmap (remaining localization work)

The infrastructure above makes any translated string render in the selected
locale. The **catalog does not yet cover the whole product** — the majority of
UI strings are still hardcoded English in JSX and need extraction into the
catalog over successive PRs. Known un-extracted surfaces, roughly in
priority order:

- **Entity detail + client islands** — the large per-entity surfaces
  (controls/risks/vendors/assets/policies/tasks/incidents detail pages, the
  `*Client.tsx` islands behind the already-translated list-page wrappers).
- **Admin area** — most `admin/**` pages (members, roles, RBAC, SSO, SCIM,
  billing, security) beyond the two already wired.
- **Toast / notification messages** — the `sonner` success/error strings
  scattered across mutation flows.
- **Zod validation messages** — the hardcoded `.min(1, '…')` / `.email('…')`
  messages in `src/app-layer/schemas/**` + `src/lib/schemas/**`.
- **User-facing error-class messages** — the English strings passed to
  `forbidden()` / `badRequest()` / `notFound()` in `src/lib/errors/**` and
  call sites.
- **Email templates** — `src/lib/email/**`, `src/app-layer/notifications/**`.
- **PDF / export text** — `src/lib/pdf/**` and the report/SoA/audit-pack
  generators.
- **Locale-aware date & number formatting** — `src/lib/format-date.ts` pins
  `en-GB` + `UTC` (deliberately, for SSR hydration safety); making it follow
  the active locale needs a hydration-safe design (thread the request locale
  through next-intl's formatter rather than reading a machine locale).

A future coverage ratchet (mirroring `sanitize-rich-text-coverage.test.ts`)
could lock a floor of catalog-wired surfaces so new pages must use next-intl;
it is intentionally not imposed yet while the initial extraction is in flight.
